## Acceptance test drift: first comparison run

After committing the curated Anthropic+Claude baseline (`f794adf`) and adding baseline assertions to `pipeline/tests/test_acceptance.py`, a manual `dr onboard Anthropic` + `dr onboard Claude` run was executed. The acceptance test caught real drift between baseline and re-onboard output.

### Findings

**1. `ewew` typo template generates a duplicate claim.**

`research/templates.yaml` has an active company template with slug `ewew` and text `COMPANY donates to environmental causes`. This is a typo/placeholder slug — the correct slug should be `donates-to-environmental-causes` (an existing claim file already uses that proper slug at `research/claims/anthropic/donates-to-environmental-causes.md`).

When `dr onboard` ran, it dispatched against the broken slug and wrote `research/claims/anthropic/ewew.md` + `ewew.audit.yaml` — an orphaned duplicate of the existing `donates-to-environmental-causes` claim.

**2. Verdict/category/confidence drift on existing anthropic claims.**

Three of the four committed anthropic claim `.md` files were rewritten by `dr onboard` with shifted fields:

| Claim | Field | Baseline | Post-onboard |
|---|---|---|---|
| `corporate-structure` | verdict | `true` | `mostly-true` |
| `publishes-sustainability-report` | category | `industry-analysis` | `ai-safety` |
| `publishes-sustainability-report` | confidence | `high` | `low` |
| `donates-to-ai-safety` | (no shift) | — | — |

Note: category drift is surprising because templates carry a fixed category — the analyst should not be re-deriving it. Worth investigating whether the analyst output overrides template-supplied category.

**3. Claude claims modified.**

All seven claude `.md` files are modified per `git status`. Diffs not yet inspected; same kind of drift expected.

**4. Source proliferation.**

The re-onboard generated 30+ new untracked source files under `research/sources/2024-2026/`. These are partially overlapping with the 32 sources already committed (different slugs for similar URLs).

### Decision needed

Two paths:

**A. Fix `templates.yaml`** — rename the `ewew` template's slug to `donates-to-environmental-causes`, delete `research/claims/anthropic/ewew.{md,audit.yaml}`, decide what to do with the modified existing claim files and new sources (likely revert via `git checkout research/claims/`).

**B. Accept the new state as the new baseline** — update `ANTHROPIC_CLAIM_BASELINE` in `test_acceptance.py` to include `ewew`, update verdict/category/confidence values to current state, commit modified claim files and new sources.

Recommended: **A**. `ewew` is clearly garbage and would confuse readers. Then re-run the comparison test to validate the proper slug-driven path.

### Holding for decision

Currently uncommitted:
- 10 modified `research/claims/{anthropic,claude}/*.md` files
- 10 modified `*.audit.yaml` sidecars
- 2 new files: `research/claims/anthropic/ewew.{md,audit.yaml}`

These should not be committed until the ewew decision lands.

## Process observation: category drift

Category appears to be a template-determined field (per `research/templates.yaml` and `pipeline/common/templates.py`). Yet the analyst output for `publishes-sustainability-report` changed it from `industry-analysis` to `ai-safety`. This suggests the analyst can override the category, which may or may not be intended.

If unintended: lock the category to template-supplied value during persistence (don't trust analyst output).
If intended: document the override mechanism and consider whether template is then more of a hint than a constraint.

Touches: `pipeline/analyst/agent.py`, `pipeline/orchestrator/persistence.py`, schema in `pipeline/common/models.py`.

## Quality-of-life: invoke + Python 3.14

Hit a `SystemError: buffer overflow` from invoke's stdin-handling thread on Python 3.14 when running long pytest tasks. Mitigated by adding `in_stream=False` to the three test tasks in `tasks.py`. Test results are correct; the exception was a tail-end stdin read after pytest already exited.

If the project upgrades invoke or pins a different Python, this workaround can be revisited.
