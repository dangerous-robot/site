## Release criteria checklist

The site can be tagged `v0.1.0` when all of these are true:

- [ ] "Standards" ŌåÆ "Criteria" rename live (display + schema + route)
- [ ] `status` field implemented and all existing claims backfilled to `published`
- [ ] Curated claim set published: Earthly Insight removed, TreadLightlyAI COI claims archived, ChatGPT description filled in
- [ ] `existential-safety-score` moved to industry scope or annotated clearly
- [ ] COI disclosure on FAQ or `/methodology` page
- [ ] GitHub repo correctly linked in footer
- [ ] README expanded to be minimally useful to an external visitor
- [ ] At least one GitHub issue template live (source submission or claim request)
- [ ] No `unverified` + `confidence: low` claims published for the operator's own company without disclosure

v0.1.0 is **not** blocked by: audit trail (4.8), public feedback forms (6.x), pipeline perf (4.6), pipeline diagram, or multiple external reviewers. `dr lint` Phase 2 (CI) is not a blocker, but Phase 1 (local command) is.

## 2. Claim status / release flag

**Status**: `plan needed`  
**Depends on**: Backfill script must run before schema lands in CI.

Add `status: z.enum(['draft', 'published', 'archived']).default('draft')` to the claims schema. Extend verdict enum with `'not-applicable'` for claims where a criterion doesn't apply to an entity.

**Build behavior**: public list pages and entity pages show only `status: published`. Direct URLs show all statuses with an appropriate notice.

**Pipeline**: `dr onboard` writes `status: draft`. `dr review` promotes to `published` and writes the audit sidecar (Phase 4.8).

**Backfill**: 84 existing claim files need `status: published` inserted. Script recommended over manual edit.

**Decision needed**: Zod default `draft` (pipeline writes it explicitly, backfill script needed) vs. default `published` (simpler backfill, pipeline must write `draft` explicitly). Recommend `draft` ŌĆö safer production posture.

## 8. Semantic versioning

**Status**: `decision needed`

**Proposal**: `package.json` version field + git tags. `v0.1.0` is the first public release tag. DNS resolution is explicitly not part of the tag criteria ŌĆö the tag records content/code readiness, not domain availability.

**Version semantics for a content site**:
- **Major**: breaking change to verdict enum, criterion definitions, or entity URL structure
- **Minor**: new entity type, new criterion, new phase shipped (e.g., Phase 4.8, Phase 6)
- **Patch**: content corrections, verdict updates, bug fixes, source additions

**Decision needed**: Display version in site footer? (Low effort: `import pkg from 'package.json'; {pkg.version}` in `Base.astro`.)
