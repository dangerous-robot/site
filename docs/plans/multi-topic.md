# Multi-topic on criteria and claims (rename + regenerate)

**Status**: Ready
**v1.0.0 surface**: full scope (cardinality change is atomic; cannot half-ship)
**Last updated**: 2026-04-26

Schema cardinality change AND field rename: `category: z.enum(...)` (single value) becomes `topics: z.array(z.enum(...))` (multi value), on both `Criterion` and `Claim`. Operator decisions 2026-04-26:

- Multi-topic is required for v1.0.0 (release blocker).
- Rename the field `category` → `topics` to align schema with the route (`/topics`) and the operator-facing vocabulary.
- Do not backfill. Existing claim and criterion content will be regenerated by re-running the pipeline against the new schema.

These three decisions reduce the work compared to a backfill plan: no mechanical content rewrites, no atomic schema-+-content discipline. The schema change lands; the pipeline output code is updated to write the new shape; existing content is deleted and regenerated.

## V1.0.0 minimum scope

This is the full scope; cardinality changes cannot be partially shipped.

### Schema (`src/content.config.ts`)

- **Criteria collection** (current `category` declaration at line 191): replace `category: z.enum([...8 values...])` with `topics: z.array(z.enum([...8 values...])).min(1).max(3)`.
- **Claims collection** (current `category` declaration at line 138): same replacement. Claim's `topics` is independent of the criterion's; convention is to mirror, but the schema does not enforce.
- The 8 enum values are unchanged: `ai-safety, environmental-impact, product-comparison, consumer-guide, ai-literacy, data-privacy, industry-analysis, regulation-policy`.

### Pydantic mirror (`pipeline/common/models.py`)

- The `Category` enum (lines 27–35) keeps its values; the field name on each model that uses it changes.
- `VerdictAssessment.category` in `pipeline/analyst/agent.py:34` becomes `topics: list[Category]`.
- Add `min_length=1` and `max_length=3` constraints on the list field via `pydantic.Field`.
- Note: criteria are loaded from `research/templates.yaml` (not `.md` files), so there is no `Criterion` Pydantic class today. Either add one for symmetry, or update the YAML loader and downstream consumers to expect `topics:` arrays. Implementer's call; document either way.

### Pipeline output (`pipeline/orchestrator/persistence.py`)

- `_write_claim_file` (frontmatter dict at lines 135–144): write `"topics": [...]` instead of `"category": ...`.
- Default behavior: claim inherits the source criterion's full `topics` set on creation. Operator can hand-edit a claim's `topics` post-pipeline if specialization is needed.
- `serialize_frontmatter` will need to emit YAML lists; verify list serialization is stable (single-line `[a, b]` vs multi-line `\n- a\n- b`); pick one and stick to it for diff-stability.

### Content regeneration (replaces backfill)

Order matters. Acceptance tests are out of scope per operator (see § Acceptance test baseline above); a manual baseline capture replaces the test as the regression reference.

1. Capture the current verdict / confidence / sources for each claim from the existing `research/claims/**/*.md` files. This is the manual regenerated-content baseline (a small JSON or markdown file kept locally during the session).
2. Land schema + Pydantic + pipeline-output changes on a branch. Build will be broken until step 5 completes (schema rejects existing `category:` content).
3. Delete existing `research/claims/**/*.md` and their `.audit.yaml` sidecars. Criteria live in `research/templates.yaml` (one file); update its records in place to use `topics: [...]` rather than deleting.
4. Re-run pipeline entrypoints (`dr onboard` for entities; any other onboarding) against the existing entities and the updated `templates.yaml`. Pipeline writes `topics: [...]` per the new schema.
5. Manual spot-check: compare regenerated content's verdict / confidence / source count against the baseline captured in step 1. Treat material drift as a rerun-or-investigate signal, not auto-fail.
6. Run `inv build` (Astro build); confirm green. This catches schema validation across the corpus.
7. Commit regenerated content + schema + pipeline + Pydantic changes in one PR (or two paired PRs landed atomically).

## Rollback

Three rollback layers, in order of cost:

1. **Pre-commit**: revert the working branch. Existing committed content is untouched if step 3 (deletion) is staged in the same commit as the regenerated content.
2. **Post-merge, pre-publish**: revert the merge commit. Schema + Pydantic + page filters + content all return together. (Acceptance tests are out of scope per operator decision; no acceptance-test gate either way.)
3. **Post-publish, on-disk content drift**: if regenerated content is published and later found to be wrong, the operator can either (a) hand-edit a published claim's `topics` array (it's just YAML), or (b) re-run the affected entity through `dr onboard` (overwrites the claim files).

Critical: keep the captured baseline (step 1 of regeneration) until the regenerated content is verified, not just for spot-checks but as the manual rollback reference.
