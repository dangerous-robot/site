## Context

A planning conversation on 2026-04-25 produced a settled set of vocab and workflow choices that need to land in the canonical docs before further v1 work proceeds. The deliverable is documentation (paragraphs, tables, lifecycle states), plus a few small plan stubs for the implementation work that follows. Implementation of new agents, new states, and threshold logic stays out of scope here; this lands the language so the implementation can be planned separately.

Decisions confirmed (canonical wording lives in `## Canonical paragraphs` below; this list is the index):

- Two canonical workflow paragraphs land: generalized vision + v1-specific (see Canonical paragraphs).
- Internal design principle "Small decisions, small models." adopted; public `/values` variant deferred (see Canonical paragraphs).
- Four object types locked: criterion, entity, source, claim (claim = output-only; sectors exist but not yet an intake).
- Object roles split intake / artifact / output. Sources are the dual-role bridge type. The split is what justifies a routing tier (see Open question).
- Four-state general lifecycle: proposed / active / flagged / retired (applies to criteria, entities, sources).
- Updated claim lifecycle: queued → in-progress (with `phase`, formerly `current_step`) → drafted → published / archived; **blocked** is a side branch from in-progress.
- Min-sources-to-proceed threshold: ≥ 2 usable sources after ingestion; below threshold, claim moves to **blocked**.
- New roles **Orchestrator** (lifecycle/state-machine) + **Router** (dispatch/small-classification). Item 3 resolved to (a) split (see `## Item 3 decision`). Router owns: source matching, kind classification, host sniff, duplicate detection, threshold-trigger to blocked, stale-claim flagging.
- Intake phrasing is net-new wording: "generates a claim for each entity it applies to" / "generates a claim for each active criterion" (no prior phrase to replace).
- **Evaluation is open-loop in v1** (evaluator-optimizer pattern, Anthropic *Building effective agents* Dec 2024; Zheng et al. *Judging LLM-as-a-Judge* NeurIPS 2023). Disagreements surface to operator; closed-loop is future-plan candidate.
- **Small-model tier is structural, not stylistic** (cascade policy; FrugalGPT, Chen et al. 2023). Router runs on smallest model class; only Analyst and Evaluator may invoke frontier models. Testable design intent.

## Object roles

Sources are dual-role; criteria and entities are intake-only; claims are output-only. This is the canonical type table for the plan:

| Type | Enters as intake? | Produced as artifact? | Published as output? |
|---|---|---|---|
| Criterion | yes | no | no |
| Entity | yes | no | no |
| Source | yes | yes (Ingestor) | no |
| Claim | no | yes (Analyst + Evaluator) | yes |

## Canonical paragraphs (source of truth for this plan)

### Generalized (vision)

> The system tracks four object types: **criteria** (reusable claim templates), **entities** (companies, products, sectors), **sources** (citable references, which can enter from outside or be produced inside the pipeline), and **claims** (verdicts about entities, generated from the other three; claims are the only output type). Anything to investigate enters a queue. Agents match incoming work to relevant items, gather and archive sources, propose a draft verdict, and evaluate it independently (the evaluation is open-loop: disagreements surface to the operator rather than being auto-resolved). The operator reviews and publishes the combined verdict, archives it, or sends it back for rework. Every step is recorded and reproducible. Small decisions are made by small models; large models are used only when the task demands it.

### V1-specific

> The system tracks four object types: **criteria** (reusable claim templates, intake-only), **entities** (companies and products in v1; sectors exist but aren't yet an intake; intake-only), **sources** (citable references; dual-role: they can enter from outside as operator-supplied intake **or** be produced inside the pipeline by the Ingestor), and **claims** (the system's only output type, never an intake). The **Orchestrator** owns each claim's lifecycle (queue → phases → drafted / blocked); the **Router** dispatches small classifications and matches incoming sources to criteria or claims. Work enters when one of the three non-claim objects is added to the queue: a **criterion** generates a claim for each entity it applies to; a **company** or **product** generates a claim for each active criterion; a **source** is matched by the **Router** to existing criteria (queuing new claim work) or to existing claims (queuing reassessment). The **Researcher** gathers candidate sources; the **Ingestor** archives each as a source file; the **Analyst** proposes a **draft verdict** and narrative once enough usable sources are available; the **Evaluator** produces an **independent evaluation** (open-loop: disagreements surface to the operator and are not auto-resolved, so the Evaluator does not feed back into the Analyst in v1). The combined post-review artifact is the **verdict**. The operator runs `dr review --approve` to publish, `--archive` to retire, or leaves a claim as draft for rework. **Citation check** is a static CI verification, not an agent.

### Design principle (internal)

> **Small decisions, small models.** Each agent is scoped to make the smallest defensible decision in its lane. Larger models are reserved for tasks that genuinely require them. The Router runs on the smallest model class; only Analyst and Evaluator may invoke frontier models. This minimizes wasteful use of large models, the central cost the project exists to make visible, and keeps behavior auditable and legible. (This is testable design intent: a future check can verify model classes per agent.)

### Design principle (public, for `/values`)

> We design our agents to make small, well-defined decisions so most work can be done by small models. Routing and classification tasks always run on the smallest model class; frontier models are reserved for analysis and evaluation. Large language models are wasteful by default; we use them only when the task demands it. The environmental discipline we ask of others, applied to our own pipeline.
